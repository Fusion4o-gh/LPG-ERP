<objective>
Implement inter-warehouse cylinder transfers with atomic dual stock entry (OUT from source, IN to destination) in a single Prisma transaction, plus cancel/reversal workflow.

Purpose: Enable warehouse managers to move cylinders between warehouse locations with full stock ledger integrity. The transfer creates paired stock entries that preserve the append-only invariant and scoped balanceAfter per location.

Output:
- warehouse-transfer.ts service (create, list, getById, cancel)
- API routes for transfers CRUD + cancel
- WarehouseTransferForm and WarehouseTransferList components
- document-numbers prefix for transfers
- Navigation entries
</objective>

    ```typescript
    type CreateTransferInput = {
      companyId: string;
      financialYearId: string;
      userId: string;
      documentNo: string;
      transferDate: string | Date;
      sourceLocationId: string;
      destinationLocationId: string;
      lines: Array<{
        itemId: string;
        cylinderState: CylinderState;
        quantity: number;
        remarks?: string;
      }>;
    };
    ```
    Implementation:
    - Validate sourceLocationId !== destinationLocationId (throw if same)
    - Validate all quantities are positive integers
    - For FILLED lines: call `assertFilledStockAvailable` with locationId = sourceLocationId (skip for EMPTY lines — empty cylinder transfers don't need filled stock check, but do validate balance after OUT entry via stockLedgerEntry's built-in check if stockAvailableCheck is enabled)
    - Create the `WarehouseTransfer` document with lines
    - For each line: create TWO stock ledger entries via `createStockLedgerEntry` (the modified one from Plan 1):
      1. OUT entry: `locationId: input.sourceLocationId, direction: OUT, sourceType: WAREHOUSE_TRANSFER, sourceId: input.documentNo, quantity: line.quantity`
      2. IN entry: `locationId: input.destinationLocationId, direction: IN, sourceType: WAREHOUSE_TRANSFER, sourceId: input.documentNo, quantity: line.quantity`
    - Write audit log with entityType "WarehouseTransfer" and entityId = documentNo
    - Return the created transfer (with lines included)

    **Function `listWarehouseTransfers(tx, input)`:**
    ```typescript
    type ListTransfersInput = {
      companyId: string;
      financialYearId: string;
      from?: string | Date;
      to?: string | Date;
      status?: TransferStatus;
      limit?: number;
    };
    ```
    Implementation: Query WarehouseTransfer with optional date range, status filter, ordered by createdAt desc, limit 50 default. Include lines count.

    **Function `getWarehouseTransferById(tx, input)`:**
    ```typescript
    type GetTransferInput = {
      companyId: string;
      id: string;
    };
    ```
    Implementation: Find by id + companyId, include lines. Throw notFound if missing.

    **Function `cancelWarehouseTransfer(tx, input)`:**
    ```typescript
    type CancelTransferInput = {
      companyId: string;
      financialYearId: string;
      userId: string;
      id: string;
      remarks?: string;
    };
    ```
    Implementation per WH-03-AC7 (cancel reverts entries):
    - Fetch the transfer with lines. Validate status is COMPLETED (can only cancel completed transfers).
    - For each line, create REVERSE entries:
      1. IN entry at source location: `locationId: transfer.sourceLocationId, direction: IN, sourceType: WAREHOUSE_TRANSFER, sourceId: transfer.documentNo, quantity: line.quantity, remarks: "Transfer cancelled"`
      (This puts stock back at source)
      2. OUT entry at destination location: `locationId: transfer.destinationLocationId, direction: OUT, sourceType: WAREHOUSE_TRANSFER, sourceId: transfer.documentNo, quantity: line.quantity, remarks: "Transfer cancelled"`
      (This removes stock from destination)
    - Update transfer status to CANCELLED.
    - Write audit log.

    **3. Create API route files:**

    `src/app/api/inventory/warehouse-transfers/route.ts`:
    - GET: Calls `listWarehouseTransfers` with query params (from, to, status, limit). Returns `{ transfers }`.
    - POST: Validates body fields (dateField for transferDate, stringField for sourceLocationId, destinationLocationId, arrayField for lines with itemId, cylinderState, quantity). Generates documentNo via `nextDocumentNumberInTransaction` inside the transaction. Calls `prisma.$transaction(...)` wrapping `createWarehouseTransfer`. Returns `{ documentNo, transfer }`.

    `src/app/api/inventory/warehouse-transfers/[id]/route.ts`:
    - GET: Calls `getWarehouseTransferById`. Returns `{ transfer }`.

    `src/app/api/inventory/warehouse-transfers/cancel/route.ts` (or POST `[id]/cancel` — use `src/app/api/inventory/warehouse-transfers/[id]/cancel/route.ts`):
    - POST: Validates body (optional remarks). Calls `cancelWarehouseTransfer`. Returns `{ transfer }`.

    All handlers: wrap in try/catch, use `getRequestContext(request)`, return `ok()`/`fail()`/`serviceError()`.

    Critical (per D-01, WH-03-AC3, WH-03-AC4, WH-03-AC5):
    - The OUT + IN pair MUST be in a single `prisma.$transaction(...)` call. If either fails, both roll back.
    - Stock source type is `WH-03-AC5` mentions atomic, use `StockSourceType.WAREHOUSE_TRANSFER` for both entries.
    - The transfer document IS the sourceId for both entries — this links them on the ledger.
    - Cancel creates reversal entries (not UPDATE/DELETE), preserving append-only invariant per WH-03-AC7.
    - No accounting voucher is created (warehouse transfers are stock-only, per architecture decision D-02 in ARCHITECTURE.md).
    - All queries scope by companyId for tenant isolation.
