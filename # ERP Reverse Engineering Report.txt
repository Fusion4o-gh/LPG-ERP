# ERP Reverse Engineering Report
### Hasnan Traders LPG Distribution ERP — Full Rebuild Plan for Codex
**Prepared:** May 6, 2026 | **Analyst:** Autonomous Agent | **Based on:** Complete live system exploration

***

## 1. Executive Summary

Hasnan Traders operates a **custom, single-tenant, LPG (Liquefied Petroleum Gas) cylinder distribution ERP** built by Evision System, Lahore (©2018–2019). The system manages the full LPG distribution lifecycle: purchasing filled cylinders from gas companies → issuing to ~259 retail customers on credit → tracking empty cylinder returns → collecting cash/bank payments → maintaining a Chart of Accounts.

The system is **functionally active** (14,618+ sale issues, 6,269+ cylinder returns, 2,651+ cash payment vouchers, 1,307+ bank receipts) but is built on a dead technology stack with no API, no mobile support, no audit log, no financial-year progression, and a day-closing function not run in 3+ years. At least 4 of 7 Sale/Purchase modules and 3 of 5 Return modules are entirely unused.

**Verdict: Full rebuild is required.** The system cannot be wrapped or incrementally patched — its architecture prevents API-first design, role-based access, multi-year accounting, and mobile access.

## 3. Core Workflows

### 3.1 Purchase Workflow (Inbound Stock)

```
TRIGGER: Gas company delivers filled cylinders

STEP 1 — Create Purchase GIRN (/DirectGIRN/add_direct_girn)
  Inputs:
    - Vendor (dropdown → 31 LPG companies)
    - Date
    - Remarks
    - 11.8 KG global price (overrides line-item price for this size)
  
  Line items (per cylinder type):
    - Item (e.g. 45.4-PSO, 11.8-Khass Gas)
    - Type: Filled or Empty
    - Received Qty
    - Unit Price
    - GST%
    - Empty Return (empties returned to vendor simultaneously)
    - Empty Stock (resulting empty stock after transaction)
    - GST Amount, Ex-GST Amount, Inc-GST Amount (calculated)

BUSINESS RULES:
  - Empty cylinders are returned to vendor WITHIN the same purchase transaction
  - GST is calculated per line item and stored separately
  - 11.8 KG price is a global override affecting all 11.8kg items in the GIRN
  - Stock increases by Recv Qty (filled)
  - Stock decreases by Empty Return qty

OUTPUT:
  - Receipt # (sequential, e.g. 1615)
  - Vendor ledger Dr (payable created)
  - Stock ledger updated (filled +, empty -)
  - GST payable recorded
```

### 3.2 Sale Workflow — Single Sale

```
TRIGGER: Customer orders LPG cylinders

STEP 1 — Create Sale (/SaleLPG/add_sale_lpg)
  Inputs:
    - Customer (dropdown, shows outstanding balance inline)
    - Date
    - Sale Type: Direct
    - Remarks
    - 11.8 KG Price (global for this sale)
    - Language: English / Urdu (invoice print language)
  
  Line items (per cylinder type):
    - Item
    - Sale Qty
    - Unit Price
    - GST (%)
    - Security (deposit per cylinder - Rs. per unit)
    - Empty Cylinder Type (what empty they return now)
    - Return Qty (empties returned AT time of delivery)
    - Filled Stock (stock available — real-time check)
    - GST Amount, Ex-GST Amount, Inc-GST Amount

BUSINESS RULES:
  - Customer can return empty cylinders at the same time as receiving filled ones
  - Security deposit is charged per cylinder (typically once per new cylinder)
  - Filled stock check prevents over-selling (Company Setting: Stock Available Check = ON)
  - Issue # is globally sequential (up to 14,618+)
  - Customer receivable (Dr) and Sales Revenue (Cr) are posted to CoA

OUTPUT:
  - Issue # assigned
  - Customer ledger Dr (receivable)
  - Sales account Cr
  - Stock: Filled -, Empty + (for returned empties)
  - Security ledger Cr (if applicable)
  - Printable invoice (English or Urdu)
```

### 3.3 Sale Workflow — Complete Day Sale (Batch)

```
TRIGGER: End of day or dispatch run — multiple customers in one session

STEP 1 — Batch entry form (/SaleLPG/add_sale_lpg_new)
  Per row (one row = one customer):
    - Customer (dropdown)
    - 11.8 KG Price
    - Item 1 / Item 2 / Item 3 (up to 3 items per customer)
    - Qty per item (inline)
    - Total (auto-calculated)
    - Payment Type: Cash or Credit
    - Amount Received (if Cash, full or partial)

  Header:
    - Date (single date for all rows)
    - Remarks

BUSINESS RULES:
  - Each row generates a separate Sale Issue # internally
  - "Cash" type immediately creates a Cash Receipt entry
  - "Credit" type leaves balance open in customer receivable
  - Stock availability shown inline per item
  - Limited to 3 item types per customer per session

OUTPUT:
  - Multiple Issue #s generated (one per row)
  - All customers' ledgers updated
  - Cash receipts auto-generated for cash sales
```

### 3.4 Cylinder Return Workflow

```
TRIGGER: Customer returns empty cylinders (separate transaction from sale)

STEP 1 — Cylinder Return (/Salereturn/add_sale_return)
  Inputs:
    - Customer
    - Date
    - Items: Empty / Filled flag
    - Return Qty per item
    - Total Amount (usually Rs.0 for empty returns, non-zero for filled returns)

BUSINESS RULES:
  - Return # is globally sequential (up to 6,269+)
  - Empty returns increase shop's empty stock
  - Filled returns (customer returning unused filled cylinders) reverse a sale
  - No approval required — operator enters directly
  - Customer's outstanding cylinder balance (Access Cylinders report) decreases

OUTPUT:
  - Return # assigned
  - Empty stock +
  - Customer stock ledger updated
  - For filled returns: customer receivable Cr, Sales Cr (reversal)
```

### 3.5 Payment & Receipt Workflows

```
CASH RECEIPT (/Multiplecashreceipt):
  - Voucher format: 1-CR-XXXX
  - Customer pays cash → Cash in Hand Dr, Customer Receivable Cr
  - Daily operation (2,145+ vouchers)
  - Multi-line: can receive from multiple accounts per voucher

CASH PAYMENT (/Multiplecashpayment):
  - Voucher format: 1-CP-XXXX
  - Company pays vendor → Vendor Payable Dr, Cash in Hand Cr
  - OR pays expenses → Expense account Dr, Cash in Hand Cr
  - 2,651+ vouchers, active daily

BANK PAYMENT (/Account_bank_payment — type BP):
  - Voucher format: 1-BP-XXXX
  - Large payments (Rs.144,000–472,000 typically)
  - Bank account Cr, Vendor/Expense Dr
  - Primarily HBL account used

BANK RECEIPT (/Account_bank_payment — type BR):
  - Voucher format: 1-BR-XXXX
  - Customer deposits into company bank
  - Bank account Dr, Customer Receivable Cr

SECURITY RECEIPT (/SecurityReceipt):
  - Customer pays cylinder security deposit
  - Fields: Customer, Cylinder type, Quantity, Amount
  - Security Liability Cr, Cash Dr
  - Rarely used in current period
```

### 3.7 Cylinder Accountability (Domain-Critical Logic)

```
THE CORE PROBLEM THIS ERP SOLVES:
  Cylinders are expensive physical assets. When sold:
  - Customer takes FILLED cylinder
  - Customer owes: Payment for gas + return of EMPTY cylinder
  
  Two parallel ledgers per customer:
  1. CASH LEDGER: Rs. owed for gas value
  2. CYLINDER LEDGER: Number of cylinders outstanding (not yet returned)

CYLINDER LIFECYCLE:
  Purchase Filled → Shop Stock (Filled)
       ↓ Sale
  Customer (Filled) → Customer has cylinder + owes empty
       ↓ Return
  Shop Stock (Empty) → Next purchase cycle
       ↓ To vendor
  Vendor gets empties back WITH next filled purchase

TRACKING MECHANISM:
  - Access Cylinders Report (Customers): shows who has how many cylinders outstanding
  - Access Cylinders Report (Own Business): shows own filled + empty stock
  - Customer Stock Ledger: cylinder movement history per customer
  - Daily Activity Report: cross-sectional view of all movements
```
